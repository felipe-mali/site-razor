# Plano de Correção - Site Razor Industrial

## Problemas Identificados

### 1. Permissões mostrando "undefined" (CRÍTICO)
**Causa raiz:** Inconsistência na estrutura de dados.
- Usuários originais no `usuarios.json` têm permissões como propriedades **flat** (top-level): `{ "admin": { "pode_ver_funcionario": true, ... } }`
- Usuários criados/atualizados via API têm permissões **aninhadas**: `{ "admin": { "permissoes": { "pode_ver_funcionario": true } } }`
- A tabela de permissões (`permissoes.js:43-45`) lê `u.pode_ver_funcionario` (flat), mas para usuários da API está em `u.permissoes.pode_ver_funcionario`

### 2. Cancelamento de orçamento - autorização incorreta
**Causa raiz:** 
- `server.js:464`: POST só permite `cargo === 'vendedor'` (deveria permitir vendedores E pode_gerenciar_permissoes)
- `cancelamentos.js:41`: Renderização só mostra para vendedores (deveria mostrar para vendedores E pode_gerenciar_permissoes)
- DELETE não tem verificação server-side (deveria exigir pode_gerenciar_permissoes)

### 3. CRM não carrega
**Causa raiz:** CRM usa IndexedDB (`razor-crm`) que é armazenamento local do navegador. Não há persistência server-side. Se o navegador limpa dados ou muda de dispositivo, os dados somem.

### 4. Chaves PIX não carregam
**Causa raiz:** `server.js:558` usa `requireAdmin` que verifica `pode_gerenciar_permissoes`. Se o usuário tem cargo 'admin' mas as permissões estão aninhadas (problema #1), o check `req.user.pode_gerenciar_permissoes` retorna `undefined`/falsy.

### 5. Sistema travado ao navegar entre abas
**Causa raiz:** `funcionario.js:139`: `trocarTela()` chama `salvarEstado()` a cada troca, que faz `JSON.stringify` + `localStorage.setItem` de todo o estado. Não é pesado por si só, mas combinado com operações assíncronas sem feedback visual dá sensação de travamento.

## Plano de Correção

### Fix 1: Normalizar estrutura de permissões (server.js + usuarios.json)

**1.0 Normalizar `data/usuarios.json` existente:** Usuários que foram criados/atualizados via API podem ter a estrutura aninhada `{ permissoes: { ... } }`. Precisamos migrar para flat. Adicionar uma função de bootstrap no `server.js` que roda uma vez no startup:

```javascript
// Normalizar permissões aninhadas para flat (executa uma vez)
function normalizarUsuarios() {
  const users = JSON.parse(fs.readFileSync('data/usuarios.json', 'utf8'));
  let mudou = false;
  for (const [key, val] of Object.entries(users)) {
    if (val.permissoes && typeof val.permissoes === 'object') {
      const p = val.permissoes;
      val.pode_ver_funcionario = p.pode_ver_funcionario ?? false;
      val.pode_ver_imagens = p.pode_ver_imagens ?? false;
      val.pode_editar_imagens = p.pode_editar_imagens ?? false;
      val.pode_gerenciar_permissoes = p.pode_gerenciar_permissoes ?? false;
      delete val.permissoes;
      mudou = true;
    }
  }
  if (mudou) {
    fs.writeFileSync('data/usuarios.json', JSON.stringify(users, null, 2));
    console.log('Permissões de usuários normalizadas.');
  }
}
normalizarUsuarios();
```

**Linha 102-118 (PUT /api/usuarios):** Salvar como flat properties:

```javascript
app.put('/api/usuarios/:id', authenticate, requireAdmin, (req, res) => {
  const users = JSON.parse(fs.readFileSync('data/usuarios.json', 'utf8'));
  const { nome, senha, cargo, ativo, permissoes } = req.body;
  if (users[req.params.id]) {
    const p = permissoes || {};
    users[req.params.id] = {
      ...users[req.params.id],
      ...(nome && { nome }),
      ...(senha && { senha }),
      ...(cargo && { cargo }),
      ...(ativo !== undefined && { ativo }),
      // Permissões sempre flat
      ...(permissoes && {
        pode_ver_funcionario: p.pode_ver_funcionario ?? false,
        pode_ver_imagens: p.pode_ver_imagens ?? false,
        pode_editar_imagens: p.pode_editar_imagens ?? false,
        pode_gerenciar_permissoes: p.pode_gerenciar_permissoes ?? false
      })
    };
    fs.writeFileSync('data/usuarios.json', JSON.stringify(users, null, 2));
    res.json({ success: true });
  } else {
    res.status(404).json({ error: 'Usuário não encontrado' });
  }
});
```

### Fix 3: Migrar CRM para server-side

**Decisão do usuário:** Migrar para persistência server-side.

**3.1 Criar arquivo de dados:** `data/crm.json` (array vazio `[]`)

**3.2 Adicionar funções helper em `server.js` (após seção de chaves pagamento):**

```javascript
const CRM_PATH = path.join(__dirname, 'data', 'crm.json');

function carregarCRM() {
  try {
    if (!fs.existsSync(CRM_PATH)) {
      fs.writeFileSync(CRM_PATH, '[]');
      return [];
    }
    return JSON.parse(fs.readFileSync(CRM_PATH, 'utf8'));
  } catch {
    return [];
  }
}

function salvarCRM(lista) {
  fs.writeFileSync(CRM_PATH, JSON.stringify(lista, null, 2));
}
```

**3.3 Adicionar 4 endpoints em `server.js`:**

- `GET /api/crm` — listar clientes (autenticado)
- `POST /api/crm` — criar cliente (autenticado)
- `PUT /api/crm/:id` — atualizar cliente (autenticado)
- `DELETE /api/crm/:id` — excluir cliente (autenticado)

Todos usam `authenticate` middleware. Cada cliente tem: `id, nome, empresa, telefone, email, vendedor, observacoes, dataCriacao, dataAtualizacao`.

**3.4 Modificar `public/js/crm.js`:**
- Remover funções IndexedDB (`abrirDBCRM`, `crmCarregarClientes`, `crmSalvarCliente`, `crmExcluirCliente`)
- Substituir por chamadas à API usando `fetch` com `Authorization: Bearer token`
- `crmSalvarDoModal`: adicionar flag `_existente` para PUT vs POST
- Remover `setTimeout` no DOMContentLoaded (linha 298), chamar `crmRenderizar()` direto

**3.5 Migrar dados existentes (opcional):** Se houver dados no IndexedDB do navegador, podemos adicionar uma função de importação que faz upload para a API.

# Plano: Funcionalidade "Chaves Pagamento"

## Contexto

O projeto é um **Node.js/Express** (não ASP.NET Razor — "Razor" é o nome da empresa). O frontend usa HTML/CSS/JS vanilla. Dados persistem em arquivos JSON em `data/`. A autenticação usa tokens em memória via cookies/header, com middlewares `authenticate` e `requireAdmin`.

O objetivo é criar uma página admin para cadastrar chaves PIX por funcionário, com controle de acesso apenas para `cargo === 'admin'`.

---

## Arquivos a Criar

### 1. `public/chaves-pagamento.html`
Página standalone seguindo o padrão de `permissoes.html`:
- Header/navbar com link "Início", "Painel", "Permissões", "Chaves Pagamento" (ativo), "Sair"
- Título: "Gerenciar **Chaves Pagamento**"
- Barra de busca por nome
- Botão "+ Novo Funcionário"
- Tabela: Funcionário | Tipo | Chave PIX | Ações (Editar | Excluir)
- Modal de cadastro/edição com campos: Nome (obrigatório), Apelido (opcional), Tipo da Chave (select), Chave PIX (obrigatório), Salvar, Cancelar
- Na tabela, coluna "Funcionário": exibir apelido se preenchido, senão exibir nome completo
- Modal de confirmação de exclusão
- Inclui: `css/style.css`, `css/funcionario.css` (para estilos de tabela/modal), `js/chaves-pagamento.js`

### 2. `public/js/chaves-pagamento.js`
Lógica da página seguindo o padrão de `permissoes.js`:
- Verificação client-side: `user.cargo === 'admin'` (redireciona para `funcionario.html` se não)
- Funções CRUD via fetch com token:
  - `listarChaves()` → GET `/api/chaves-pagamento`
  - `adicionarChave()` → POST `/api/chaves-pagamento`
  - `editarChave(id)` → PUT `/api/chaves-pagamento/:id`
  - `removerChave(id)` → DELETE `/api/chaves-pagamento/:id`
- Validações client-side:
  - Nome obrigatório
  - Apelido: opcional (se preenchido, exibe na tabela ao invés do nome completo)
  - Chave obrigatória
  - Nome duplicado
  - Chave PIX duplicada
  - Formato da chave conforme tipo (CPF: 11 dígitos, CNPJ: 14 dígitos, Telefone: 10-11 dígitos, Email: regex básico, Aleatória: livre)
- Lógica de exibição na tabela: `apelido || funcionario` (se apelido tiver valor, usa ele; senão, usa o nome completo)
- Barra de pesquisa instantânea por nome (pesquisa tanto no campo `funcionario` quanto no `apelido`)
- Modal com `active` class toggle
- Sanitização: escape de `<`, `>`, `&`, `"`, `'` em inputs antes de renderizar na tabela

## Arquivo a Modificar

### 3. `server.js` — Novas rotas API
Adicionar na seção de rotas (antes do `app.listen`):

```
CRUD /api/chaves-pagamento:
- GET    /api/chaves-pagamento     → authenticate + requireAdmin → lista todas
- POST   /api/chaves-pagamento     → authenticate + requireAdmin → cadastra (valida duplicatas)
- PUT    /api/chaves-pagamento/:id → authenticate + requireAdmin → edita (valida duplicatas)
- DELETE /api/chaves-pagamento/:id → authenticate + requireAdmin → exclui
```

Dados em `data/pagamento_chaves.json` com estrutura:
```json
{
  "funcionarios": [
    {
      "id": "1",
      "funcionario": "João Silva",
      "apelido": "João",
      "tipo_pix": "CPF",
      "chave_pix": "12345678900",
      "created_at": "2026-07-20 14:30",
      "updated_at": "2026-07-20 14:30"
    }
  ]
}
```

Funções helper:
- `carregarChaves()` — lê JSON, cria arquivo se não existir
- `salvarChaves(lista)` — escrita segura via `writeFileSync` (mesma prática do projeto)

Validações server-side:
- Campos obrigatórios: `funcionario`, `tipo_pix`, `chave_pix`
- Campo opcional: `apelido` (trim, aceitar vazio/null)
- `tipo_pix` deve ser um dos: CPF, CNPJ, Telefone, Email, Aleatória
- Sanitização: trim + remove caracteres perigosos de `funcionario` e `apelido`
- Validação de formato conforme tipo (CPF: 11 dígitos, CNPJ: 14 dígitos, Telefone: 10-11 dígitos, Email: regex)
- Checagem de duplicatas: mesmo nome ou mesma chave_pix (ignorando o registro atual na edição)
